几个需要特别注意的设计要点：

幂等性是重中之重。 老人 App 网络不稳定，用户可能反复点"下单"或"确认"。所有写接口都要求客户端在 Header 带 X-Idempotency-Key（UUID），服务端用 Redis 做 24h 幂等缓存，相同 key 的请求直接返回第一次的结果，不重复执行。

接单的并发控制。 同一订单可能被多个服务商同时点击接单，用 Redis SET NX（分布式锁）实现，锁粒度是 order_id，持锁时间 3 秒，抢到锁才能写库，其余返回"已被接单"。

GPS 围栏签到防刷单。 checkin 接口收到坐标后，计算与老人地址的距离，超过 500m 拒绝签到并告警。这是银发服务平台区别于普通外卖平台的关键信任机制，不能省。

家属代确认。 confirm 接口需要校验调用方身份，老人本人或绑定家属都可以触发，但 24h 超时后系统自动确认也是一条路径——因为很多老人不会主动操作。

派单策略做成可插拔策略模式。 初期用"就近+评分综合排序"，高峰期可以切换为抢单模式，节假日可以动态调整权重，不需要改核心代码。

这是个值得深想的问题。表面上像美团，但本质上有几个关键差异。

相似的地方： 平台撮合模式（C2B2C）、服务商入驻、派单调度、评价体系——这些结构确实和美团一脉相承。

但核心差异在三个维度：

① 付款决策者 ≠ 使用者 美团是用户自己饿了自己点自己付钱，决策链极短。老龄平台的真实结构是：子女付钱、老人用服务、政府可能补贴一部分。这意味着你有两个客户——老人是体验端，子女是付费端，两套 App、两套运营逻辑，转化路径也完全不同。美团不需要说服用户的家人。

② 服务的非标准化程度远高于外卖 外卖是标品——一份宫保鸡丁就是一份宫保鸡丁。老龄服务里"陪诊"、"助浴"、"康复护理"每一单的实际内容差异极大，服务质量很难用评分系统完整量化，纠纷率和售后成本会远高于美团。这也是为什么签到 GPS 验证、服务凭证上传是必须的——它们本质上是在对抗非标带来的信任问题。

③ 政府是隐形的第三方 美团完全是市场化的。老龄服务在中国有一个几乎所有创业者都绕不开的现实：民政局、街道、社区是重要的流量和补贴入口。很多老人是通过社区推荐才信任一个平台的，政府采购（B2G）可以成为重要的收入来源，但同时也意味着接口要对接政务系统、数据要符合监管要求。美团从来不需要跟政府谈数据共享协议。

所以更准确的定位应该是：

"美团的撮合架构 + 京东的品控逻辑 + 政务系统的对接能力"

纯照搬美团模式的风险是：把自己做成一个低价竞争的劳务中介平台，服务商质量参差不齐，老人投诉率高，子女不信任，最后口碑崩掉。

真正的护城河不是派单算法，而是服务标准化体系——把"助浴"拆解成 12 个标准动作、把"陪诊"定义成标准流程，让每一单的质量可预期。这件事美团不需要做，但你必须做。

这张图解释了核心设计思路。关键决策是三层分离：

第一层：主体层（谁入驻）分两张表，provider_org（机构）和 provider_individual（个人），字段完全不同——机构有营业执照、对公账户、员工管理；个人有实名认证、身份证、个人银行卡。

第二层：执行层（谁干活）统一抽象成 provider_worker 一张表。机构的员工在这里注册为 worker，归属字段指向 provider_org_id；个人服务商在这里也有一条 worker 记录，归属字段指向自身。订单服务、GPS 签到、派单系统只认 worker_id，完全不知道背后是机构还是个人，逻辑统一，代码不分叉。

第三层：结算层（钱打给谁）在订单完成后才分叉——查 worker 的归属，机构 worker 的钱打到对公账户，个人 worker 的钱走个人提现。

这样设计的好处是：你加一种新的服务商类型（比如未来加政府兜底服务机构），只需要新建一张主体表，执行层和订单层完全不用改。

数据库核心表大概是这样：

sql
-- 机构主体
provider_org (org_id, name, license_no, bank_account, status, ...)

-- 个人主体  
provider_individual (individual_id, real_name, id_card, bank_card, status, ...)

-- 统一执行人（两者都映射到这里）
provider_worker (
  worker_id,
  type ENUM('org_staff', 'individual'),
  org_id NULL,        -- 机构员工填这个
  individual_id NULL, -- 个人服务商填这个
  name, phone, skills[], service_area, status, rating
)

-- 订单只存 worker_id，不管来源

-- 1. 实时位置（存 Redis，不入 MySQL）
-- Key: location:worker:{worker_id}
-- Value: Hash { lat, lng, accuracy, order_id, updated_at }
-- TTL: 600秒（10分钟无上报自动过期）

CREATE TABLE worker_location_track (

  lat          DECIMAL(10,7)  NOT NULL,         -- 纬度，精度约1cm

  created_at   TIMESTAMP      DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_order (order_id, recorded_at),     -- 按订单查轨迹
  INDEX idx_worker (worker_id, recorded_at)    -- 按人查历史
) 
PARTITION BY RANGE (UNIX_TIMESTAMP(recorded_at)) (

  PARTITION p2024_01 VALUES LESS THAN (UNIX_TIMESTAMP('2024-02-01')),
  PARTITION p2024_02 VALUES LESS THAN (UNIX_TIMESTAMP('2024-03-01'))
  -- ...

-- point_type 枚举：
-- 1 = CHECKIN       签到打卡（关键节点，永久保留）
-- 2 = TRACKING      服务中常规轨迹（90天后删）
-- 3 = CHECKOUT      完成打卡（关键节点，永久保留）
-- 4 = SOS           SOS触发位置（永久保留）

  order_id     VARCHAR(36)    NOT NULL UNIQUE, -- 一单只有一次签到

  checkin_lng  DECIMAL(10,7)  NOT NULL,

  elder_lng    DECIMAL(10,7)  NOT NULL,
  distance_m   SMALLINT       NOT NULL,        -- 签到时计算出的距离（米）
  is_valid     BOOLEAN        NOT NULL,        -- 是否通过围栏校验（<=500m）

  INDEX idx_order (order_id)

几个关键的设计决策解释一下：

DECIMAL(10,7) 存坐标而不用 FLOAT。FLOAT 精度只有 6-7 位有效数字，经纬度用 FLOAT 会有几米误差，DECIMAL(10,7) 精度到小数点后7位，误差小于1厘米，存围栏判断不会出问题。

签到坐标快照要同时存老人地址坐标。老人可能搬家或改地址，如果只存服务商坐标、事后再查老人地址重算距离，会因为地址变更导致距离不对，纠纷时无法还原当时情况。下单时把老人地址坐标快照进来，签到记录就是自闭合的。

point_type = CHECKIN/CHECKOUT/SOS 的记录永久保留，普通轨迹点90天后按分区批量删除。这样既控制了存储成本，又保留了纠纷所需的关键证据。

常规轨迹不要实时写 MySQL，先走 Kafka 攒批，每分钟批量写入一次，把高频小写变成低频批写，MySQL 压力下降 95%。

家属 App 查看实时位置直接读 Redis，不查 MySQL，延迟从秒级降到毫秒级，而且 Redis GEO 结构天然支持"附近的服务商"这类查询，不需要额外的空间索引。

service_sku 存的是平台定义的服务模板，比如"助浴服务"、"陪诊服务"，这是 catalog-service 管的，所有服务商共用。

order_dispatch 派单表是做什么的？

它记录的是派单过程本身，不是派单结果。一个订单可能经历"先派给A、A拒绝、再派给B、B接单"的过程，这张表就会有两条记录。dispatch_status 枚举如下：

PENDING   → 已推送给服务商，等待响应
ACCEPTED  → 服务商接单
REJECTED  → 服务商拒绝（含超时自动拒绝）
CANCELLED → 平台撤回此次派单（换人派）

下单时为什么要把 SKU 的名称和价格快照进 orders 表？

因为 SKU 的价格将来可能涨价或下架，但已经成交的订单必须永远保持当时的价格，不受后续改价影响。所以下单那一刻，sku_name、unit_price、duration 都要快照写入 orders，后续结算、退款、纠纷都以快照为准，不回查 SKU 表。

派单流程的数据写入顺序：

1. orders 插入一条记录，status = PENDING_ASSIGN
2. 派单算法选出候选 worker
3. order_dispatch 插入一条记录，status = PENDING
4. 推送给服务商 App
5a. 服务商接单 → dispatch.status = ACCEPTED，orders.status = ACCEPTED
5b. 服务商拒绝/超时 → dispatch.status = REJECTED → 重新执行步骤2

这样派单历史完整可追溯，纠纷时能看到当时是谁先接的、谁拒绝过。
